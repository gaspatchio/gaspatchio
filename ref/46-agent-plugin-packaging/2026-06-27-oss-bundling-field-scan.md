# Packaging Decisions — Field-Evidence Memo

Four field scans of how real OSS projects bundle Agent Skills / plugins converge on a clear set of conventions, and they let us finalize all four open packaging choices. The short version: the ecosystem ships **one canonical skill tree consumed via thin per-tool manifests**, versions by SHA at the skill layer but **semver at the plugin layer**, and only ships a router skill at high skill counts — so two of our four leans hold, one needs splitting, and one should change.

## What real repos actually do

**Skill location — flat `skills/` for catalogs, `.agents/skills/` for in-place project repos; `.github/skills/` is NOT a Copilot delivery path.**
Pure skill/marketplace repos use a plain top-level `skills/<name>/SKILL.md` (`anthropics/skills`, `vercel-labs/agent-skills`, `obra/superpowers`, `deepgram/skills`, `bmad-code-org/bmad-automator`, `github/awesome-copilot`). Project repos that embed skills for in-place use put them in a dot-dir: `flutter/devtools` ships its 7 skills **only** in `.agents/skills/`, and the `vercel-labs/skills` CLI hard-codes `.agents/skills` as its default install dir, mapping ~25 agents to it. Decisively, **GitHub Copilot's own docs say project skills load from `.github/skills`, `.claude/skills`, OR `.agents/skills`** — so `.agents/skills/` is read by Copilot natively. In the wild `.github/skills/` is used only for *internal CI/maintenance* skills (`Azure/agentops`'s `.github/skills/release-management/`, separate from its 7 product skills; `github/awesome-copilot`'s own `.github/skills/agentic-workflows/`), or as a *generated copy* in multi-tool bundles (`sampleXbro/agentsmesh`). Copilot product-skill delivery is done via a `.github/plugin/marketplace.json` manifest (`Azure/agentops`), not a skills mirror.

**Multi-tool handling — one canonical tree + manifest pointers / symlinks / generated copies; never hand-maintained duplicate trees.**
`obra/superpowers` keeps one `skills/` dir and each per-tool dir (`.codex-plugin/`, `.cursor-plugin/`, `.kimi-plugin/`, `gemini-extension.json`) holds **only** a ~10-key manifest pointing `skills: "./skills/"` at the same tree — zero duplicated SKILL.md files. `regent-vcs/re_gent` commits `.claude/skills/*` as git **symlinks** (mode 120000) back to real `.agents/skills/*` files. The `vercel-labs/skills` CLI's default mode is `--symlink` ("single source of truth"), with `--copy` as the non-default escape hatch. Where copies do exist (`agentsmesh`: `.agents/skills` 90 entries → `.claude/skills` 78 filtered), they are **generated/filtered**, not hand-written. Note the Claude vs Codex `.agents/plugins/marketplace.json` schemas genuinely **diverge** (flat `source`+`version` vs `interface{}`+`source{}`+`policy{}`, no version) — so multi-tool = maintaining N manifest schemas, not N skill copies.

**Manifest / marketplace footprint — single-repo marketplace, minimal manifests, many skills under few plugins.**
Every example is a single-repo marketplace (`source: "./"` or `"./plugins/<name>"`); nobody used a dedicated separate marketplace repo. The minimal footprint is **one `.claude-plugin/plugin.json` + one `marketplace.json`** at repo root (`bmad-automator`, `deepgram/skills`, `Azure/agentops`). Anthropic's first-party `plugin.json` is deliberately minimal — `{name, description, author}`, **no skills enumeration** (auto-discovery). `anthropics/skills` slices ONE physical tree into THREE plugins via explicit `skills: ["./skills/xlsx", ...]` arrays — i.e. many skills grouped under few named plugins, **not** one-plugin-per-skill. Skills are referenced by relative path, decoupling manifest size from skill content (`github/awesome-copilot`).

**Versioning — layered: omit/SHA at the skill level, semver at the plugin level.**
The agentskills.io spec defines **no top-level `version` field** for SKILL.md (required: `name`, `description`; a version may only live under the freeform `metadata:` map). `anthropics/skills` SKILL.md frontmatter is literally just `name`/`description`/`license` — no version. At the marketplace layer, `anthropics/claude-plugins-official` pins remote plugins by **commit SHA** (192/192 object-sources carry `sha`; only ~1/3 add a `ref`; just 14/243 carry a top-level version). But for a plugin's **own** manifest, **pinned semver is the prevailing convention where a release pipeline exists**: `obra/superpowers` 6.0.3, `Azure/agentops` 0.6.0, `bmad-automator` 1.15.0, `artibot` 4.29.0, `awesome-copilot` (generator defaults absent versions to `1.0.0`). Pure "omit-version, SHA-driven" was **not** observed for a plugin's own manifest — SHA/ref pinning is used only when *referencing external repos*.

**Generate vs hand-author — SKILL.md bodies hand-authored; manifests/versions generated from an SSOT in mature repos.**
Skill content is universally hand-written. The polished multi-tool repos generate the *manifests*: `ynab-mcp-server` treats `package.json.version` as SSOT and `sync-plugin-metadata.mjs` writes it into 12 files with a `check-release-consistency.mjs` CI guard; `Azure/agentops`'s `cut-release.sh` jq-syncs one version into both marketplace.json copies atomically; `github/awesome-copilot`'s `eng/generate-marketplace.mjs` derives the catalog from per-plugin manifests, and its `check-plugin-structure.yml` **rejects materialized skills in plugin dirs on main** (canonical files live once in `skills/`, copied in only at publish). The lighter Claude-only repos hand-maintain manifests with a smoke-test guard.

**Router / overview skill — usually filled by always-loaded AGENTS.md; a dedicated router skill appears mainly at large skill counts.**
The majority pattern is **no router skill** — routing lives in keyword-rich per-skill `description:` frontmatter plus an always-loaded `AGENTS.md`/`copilot-instructions.md` (`Azure/agentops` ships both; `awesome-copilot`, `bmad-automator`, `tabtint`, `anthropics/skills` ship task skills only). A dedicated router does exist in mature/large bundles: `obra/superpowers`'s `using-superpowers`, `agentsmesh`'s `using-agent-skills`, `ShravanSunder/ai-tools`'s `orchestrator-goal` (alongside 20 task skills), `deepgram`'s `deepgram` meta-plugin. Anthropic's multi-skill plugins route instead via an orchestrating **command** (`/plugin-dev:create-plugin`), not a skill.

## Decision table

| Choice | Field evidence | Verdict |
|---|---|---|
| **(1) `.agents/skills/` only [verify Copilot]** | Copilot's **own docs** confirm it reads `.agents/skills` natively (also `.github/skills`/`.claude/skills`); `vercel-labs/skills` CLI defaults to `.agents/skills`; `flutter/devtools` ships skills **only** in `.agents/skills/` (no `.claude/skills` at all); `laravel-agent-mcp`'s target map routes Codex/Amp/Gemini/OpenCode/Antigravity to `.agents/skills`. `.github/skills/` in the wild = CI/maintenance skills, not product delivery; Copilot product delivery is a `.github/plugin/marketplace.json` manifest, not a skills mirror. | **CONFIRM.** Copilot verified — `.agents/skills/` alone covers Claude (via plugin manifest `source:./`), Codex, and Copilot. Do NOT add `.github/skills/`. *Caveat:* native Claude Code `/plugin` discovery wants a `.claude/skills` dir or a `.claude-plugin` manifest pointing at the tree — handle that via the manifest (choice 2) or a symlink, not a copy. |
| **(2) Full per-tool manifest + marketplace set** | Evidence **contradicts** a *full* set as the baseline. Anthropic's own first-party pattern is **minimal**: one `marketplace.json` + minimal `plugin.json`, all skills under one plugin via a `skills:[...]` array, no per-skill manifests. Multi-tool repos add a **thin** sibling manifest **per ecosystem you actually target** (`superpowers`: `.codex-plugin/`, `.cursor-plugin/`, `gemini-extension.json` — ~10 keys each), pointing at the same tree. Claude-only is a legitimate, common shipping choice (`bmad-automator`, `tabtint`). Schemas diverge (Claude vs Codex/`.agents`), so each tool = a real second schema, not a copy. | **SPLIT.** Ship the minimal core now — **one `.claude-plugin/plugin.json` + one `marketplace.json`** listing all 7 skills by relative path (`anthropics/skills` model). Add per-tool manifests **only for ecosystems we actively target** (e.g. a `.agents/plugins/marketplace.json` for Codex, a `.github/plugin/marketplace.json` for Copilot) — and only behind the SSOT generator (choice 3 / `ynab`/`awesome-copilot` pattern). A blanket "full per-tool set" is over-build. |
| **(3) SHA-driven versioning** | **Contradicted at the layer that matters.** SHA pinning is real only for **referencing external repos** (`claude-plugins-official` 192/192 sources carry `sha`). For a plugin's **own** manifest, the field norm is **pinned semver** wherever a release pipeline exists (`superpowers` 6.0.3, `agentops` 0.6.0, `bmad` 1.15.0, `artibot` 4.29.0, `awesome-copilot` default `1.0.0`). At the **SKILL.md** layer, the spec has no `version` field and reference repos omit it. | **CHANGE to layered: semver on the plugin manifest, no version in SKILL.md.** Pin one semver on our `plugin.json`/`marketplace.json` (driven by an SSOT bump script — `ynab`/`agentops` pattern; semver *without* an automated bumper is the worst of both worlds), and put **no** `version` in SKILL.md frontmatter (spec-compliant; matches `anthropics/skills`). Use SHA/`ref` only if/when *external* repos reference us. |
| **(4) Lightweight router skill in v1** | Mixed. Majority of comparable repos ship **no** router skill and let always-loaded `AGENTS.md` + per-skill descriptions do the routing (`awesome-copilot`, `agentops`, `bmad`, `anthropics/skills`); we already have a "Skill Routing" table in `AGENTS.md`. Dedicated routers appear mainly at **large** skill counts (`superpowers` 14, `agentsmesh` 78–90, `ai-tools` 20). **Caveat:** Codex/Copilot don't auto-load `AGENTS.md` the way Claude Code does, so for those tools a thin router skill is the bridge. | **SPLIT / lean CHANGE.** For Claude Code, **don't** spend a skill slot — the `AGENTS.md` routing table already fills the role (majority pattern at 7 skills). Ship a thin router skill in v1 **only if** we commit to Codex/Copilot in v1, where it's the discovery bridge. If v1 is Claude-only, defer the router. Either way, keep routing canonical in `AGENTS.md` + sharp per-skill `description:` triggers. |

## Surprises / cautions

- **`.github/skills/` is a false friend.** In real repos it holds *internal CI/maintenance* skills (`Azure/agentops` release-management, `awesome-copilot` agentic-workflows), not a Copilot product mirror. Copilot reads `.agents/skills` natively and is delivered via `.github/plugin/marketplace.json`. Adding `.github/skills/` would create a maintained copy that buys nothing.
- **Anthropic's own `plugin-dev` skills violate the spec** by using a top-level `version:` key (0.1.0/0.2.0) that the spec doesn't define — don't copy that; if we ever version a skill, it goes under `metadata:`.
- **Semver without an automated bumper is the worst of both worlds.** Every repo with manifest semver pairs it with a sync/bump script (`ynab`'s 12-file sync + `check-release-consistency.mjs` guard; `agentops`'s `cut-release.sh`). Adopt the SSOT-generate-then-CI-guard discipline — this is exactly the drift our `skills-manifest-sync` branch exists to kill.
- **`anthropics/skills`' marketplace.json is hand-edited and shows it** (a stray `,` on its own line) — proof that hand-syncing manifests drifts. Generate, don't hand-edit.
- **`${CLAUDE_PLUGIN_ROOT}` is mandated** by the canonical `plugin-structure` SKILL.md for all intra-plugin references — no hardcoded/relative paths, and component dirs must NOT live inside `.claude-plugin/`. Worth enforcing in our structural guard.
- **Smart pattern — slice one tree into sub-bundles.** `anthropics/skills` publishes 3 overlapping plugins from one physical `skills/` tree via explicit path arrays; if we ever want to ship subsets (e.g. modeling vs review skills) we can do it without copying.
- **`CLAUDE.md → AGENTS.md` symlink shim is widespread** (`vercel-labs/agent-skills`, `obra/superpowers`) — validates the shim gaspatchio already uses.
- **`awesome-copilot` actively rejects materialized skills inside plugin dirs on `main`** via CI — strong precedent for "single tree, materialize copies only at publish," which is the cleanest answer to our sync problem.
