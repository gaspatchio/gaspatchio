# Release Pipeline & SBOM Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extend the `release` job in `.github/workflows/CI.yml` so a tagged release publishes to PyPI via Trusted Publishing (free PEP 740 attestations), generates and attaches a CycloneDX SBOM, runs `twine check --strict`, and pins the actions it touches — plus add a root `SECURITY.md`. Authored for the new repo `gaspatchio/gaspatchio`.

**Architecture:** All changes are confined to one workflow job plus two new files (`SECURITY.md`, a settings runbook). The PyPI publish step is guarded with `if: github.repository == 'gaspatchio/gaspatchio'` so the workflow is safe to sit in the repo before the move and only publishes from the new repo. The SBOM is generated by `anchore/sbom-action` (pinned syft) scanning the repo root *after* artifacts are downloaded into `dist/`, so it captures `Cargo.lock` (Rust), `pyproject.toml` (Python direct), and the built wheels' metadata in one pass.

**Tech Stack:** GitHub Actions, `anchore/sbom-action` (syft) → CycloneDX, `pypa/gh-action-pypi-publish` (Trusted Publishing / OIDC), `uv`/`uvx` for `twine check`, maturin-built wheels.

## Task 6: Pin the touched actions to commit SHAs

**Files:**
- Modify: `.github/workflows/CI.yml` (the `release:` job)

Pin every action the `release:` job uses to a full commit SHA with a trailing version comment (so Dependabot can still bump). The five to pin: `actions/checkout`, `astral-sh/setup-uv`, `actions/download-artifact`, `anchore/sbom-action`, `pypa/gh-action-pypi-publish`.

- [ ] **Step 1: Resolve the current commit SHA for each action's pinned tag**

Run (requires `gh` authenticated). These pin to the exact major tags introduced in Tasks 2–4 (`checkout@v4`, `setup-uv@v6`, `download-artifact@v4`, `sbom-action@v0`); `gh-action-pypi-publish` is pinned to its latest release tag (its `release/v1` branch ref can't be resolved by the commits API):
```bash
for spec in \
  "actions/checkout@v4" \
  "astral-sh/setup-uv@v6" \
  "actions/download-artifact@v4" \
  "anchore/sbom-action@v0"; do
  repo="${spec%@*}"; ref="${spec#*@}"
  echo "$repo@$(gh api "repos/$repo/commits/$ref" --jq .sha)   # $ref"
done
t=$(gh api repos/pypa/gh-action-pypi-publish/releases/latest --jq .tag_name)
echo "pypa/gh-action-pypi-publish@$(gh api "repos/pypa/gh-action-pypi-publish/commits/$t" --jq .sha)   # $t"
```
Expected: a line per action of the form `owner/repo@<40-char-sha>   # <tag>`. Use the resolved SHA (not the tag) in the workflow, keeping the `# <tag>` as a trailing comment.

- [ ] **Step 2: Replace each `uses:` in the release job with `owner/repo@<sha> # vX.Y`**

For each of the five actions, change e.g. `uses: actions/checkout@v5` to:
```yaml
        uses: actions/checkout@<sha-from-step-1>   # v5
```
Do this for all five actions in the `release:` job. Leave the other jobs' actions unpinned for now (full-workflow pinning is a fast-follow).

Run: `awk '/^  release:/,/^  [a-z]/' .github/workflows/CI.yml | grep -nE "uses: .*@(v[0-9]|release/|main|master|stable)"`
