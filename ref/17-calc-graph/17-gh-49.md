## Overview

Currently, ActuarialFrame columns must be accessed using dictionary-style syntax: `af["column_name"]`. This RFC proposes adding support for variable-based syntax to make model code more concise and natural to write, while maintaining full IDE autocomplete support.

### Current Syntax
```python
af["issue_age"] = af["Policyholder issue age"] + af["term_offset"]
af["age"] = af["issue_age"] + af["year"] - 1
```

### Proposed Syntax
```python
issue_age = policyholder_issue_age + term_offset
age = issue_age + year - 1
```

## Motivation

1. **Cleaner Code**: Reduces repetitive `af["..."]` syntax
2. **Natural Python**: Feels more like standard Python programming
3. **Less Typing**: Faster to write models
4. **IDE Support**: Must maintain autocomplete, type hints, and go-to-definition
5. **Backwards Compatible**: Both syntaxes should work together

## Proposed Implementation

### Static Code Generation Approach

Generate a Python file with column variable assignments before model execution:

```python
# model_columns.py - Auto-generated
from typing import TYPE_CHECKING
if TYPE_CHECKING:
    from gaspatchio_core import ColumnProxy

# Column variables from model-points.parquet
policy_number: ColumnProxy = af["Policy number"]
policyholder_issue_age: ColumnProxy = af["Policyholder issue age"] 
policyholder_sex: ColumnProxy = af["Policyholder sex"]
# ... etc
```

### Integration Options

1. **Automatic Generation**: Generate before each model run
2. **CLI Command**: `gspio generate-columns model-points.parquet`
3. **Watch Mode**: Auto-regenerate when model points change

### Usage Example

```python
# model_calculation.py
from gaspatchio_core import ActuarialFrame
from .model_columns import *  # Generated variables

def main(af: ActuarialFrame) -> ActuarialFrame:
    # Full IDE autocomplete works\!
    issue_age = policyholder_issue_age + term_offset
    age = issue_age + year - 1
    
    # Can still use dictionary syntax
    af["custom_column"] = age * 2
    
    return af
```

## Assumption Table Integration

The lookup method should accept both strings and ColumnProxy objects:

```python
# Current (still works)
mortality_table.lookup(
    age="age_mort_lookup",
    sex="Policyholder sex",
    smoking_status="Policyholder smoking status",
    year_lookup="year"
)

# Proposed (with variables)
mortality_table.lookup(
    age=age_mort_lookup,
    sex=policyholder_sex,
    smoking_status=policyholder_smoking_status,
    year_lookup=year
)
```

### Implementation for lookup()

```python
def lookup(self, **kwargs):
    # Convert ColumnProxy objects to column names
    resolved_kwargs = {}
    for param_name, value in kwargs.items():
        if hasattr(value, '__gaspatchio_column_name__'):
            resolved_kwargs[param_name] = value.__gaspatchio_column_name__
        elif isinstance(value, str):
            resolved_kwargs[param_name] = value
        else:
            raise TypeError(f"lookup parameter {param_name} must be a column name or ColumnProxy")
    
    return self._internal_lookup(**resolved_kwargs)
```

## Column Name Sanitization

- "Policy number" → `policy_number`
- "CSO table" → `cso_table`
- "Policyholder issue age" → `policyholder_issue_age`
- Handle Python keywords: `class` → `class_`

## Benefits

1. **Full IDE Support**: Autocomplete, type hints, go-to-definition all work
2. **Static Analysis**: Linters and type checkers work normally
3. **No Runtime Magic**: Easier to debug and understand
4. **Gradual Adoption**: Can mix both syntaxes in same model
5. **Version Control**: Generated file can be committed

## Backwards Compatibility

This feature is fully backwards compatible. Existing models using `af["column"]` syntax will continue to work unchanged. The variable syntax is opt-in.

## Alternative Approaches Considered

1. **Dynamic Variable Injection**: Using exec/globals manipulation - rejected due to loss of IDE support
2. **Context Manager**: `with af.variables():` - rejected due to IDE limitations
3. **Decorator Approach**: `@with_column_variables` - rejected for same reasons

The static code generation approach was chosen specifically to maintain full IDE support while providing cleaner syntax.
